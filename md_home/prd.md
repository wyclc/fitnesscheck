## 三、背景簡介（目標 / 背景 / 價值）

### 3.1 問題背景
目前許多人有運動意願，但常遇到以下痛點：

- 容易被日常生活、工作、課業打斷，無法持續執行
- 缺乏外部約束機制，導致目標容易中斷
- 不清楚運動後自己的身體素質或狀態有何變化
- 運動過程較枯燥，缺乏持續投入的動機

### 3.2 產品目標
打造一款以「押金機制 + 養成式寵物」為核心的運動習慣養成 App，讓使用者：

- 先透過押金建立承諾感
- 再開始設定長期運動目標
- 在執行過程中透過虛擬寵物成長獲得回饋
- 目標達成時獲得虛擬寵物收藏與押金返還
- 看見自己在運動過程中的進度與身體變化

### 3.3 產品價值

#### 對使用者的價值
- 提高開始運動後持續執行的機率
- 透過遊戲化機制降低中途放棄率
- 讓抽象的「養成習慣」變得具體、可視化
- 讓運動成果不只停留在感覺，而有明確呈現

#### 對產品驗證的價值
- 驗證「押金門檻」是否能有效提升承諾感
- 驗證「養寵物成長」是否能提升使用者完成率
- 驗證使用者是否能理解「先押金、再設定目標」的流程

## 五、使用者角色

### 5.1 主要使用者：想建立運動習慣的人
**特徵**
- 有運動意願，但執行不穩定
- 希望建立長期運動目標
- 需要外部約束與回饋機制
- 想看到自己是否有進步

**核心需求**
- 有一個明確開始的入口
- 有持續執行的動力
- 可以簡單追蹤進度
- 可以看見成果與回饋

### 5.2 虛擬角色：被養大的寵物
**角色定位**
- 非功能使用者，而是產品中的情感回饋載體
- 扮演進度可視化與獎勵化的介面角色

**產品任務**
- 將抽象的「持續運動」轉化成具體成長
- 在過程中提供陪伴感與成就感
- 在完成時成為可收藏的獎勵

## 七、功能流程

### 7.1 首次使用主流程
1. 使用者進入 App
2. 首頁顯示產品核心概念與主要 CTA
3. 使用者點擊「開始設定押金」
4. 進入押金選擇頁，選擇 100 / 200 / 300 元
5. 進入押金確認頁，模擬付款成功
6. 系統解鎖「目標設定功能」
7. 使用者建立運動目標
8. 建立完成後進入首頁 / 目標詳情頁
9. 使用者開始打卡執行目標
10. 寵物隨進度成長
11. 根據結果進入：
   - 達成：顯示完成頁，獲得寵物、押金返還
   - 未達成：顯示失敗頁，押金不返還

### 7.2 日常執行流程
1. 使用者進入首頁
2. 查看目前進行中的目標
3. 點擊進入目標詳情
4. 完成一次運動後點擊打卡
5. 系統更新：
   - 已完成次數
   - 進度條
   - 寵物成長階段
   - 身體變化摘要（mock）
6. 返回首頁看到更新後狀態

### 7.3 目標完成流程
1. 最後一次打卡後達成目標條件
2. 系統將目標狀態改為「已完成」
3. 顯示完成結果頁 / 彈窗
4. 顯示：
   - 獲得的虛擬寵物
   - 押金返還成功
   - 達成摘要
5. 寵物進入收藏頁
6. 目標進入歷史紀錄

### 7.4 目標失敗流程
1. 目標時間結束但未達成條件
2. 系統將目標狀態改為「失敗」
3. 顯示失敗頁 / 失敗狀態卡
4. 顯示：
   - 未達成原因摘要
   - 押金不返還
   - 鼓勵重新設定目標
5. 可重新開始下一輪押金設定

  - 目標詳情

## 九、頁面設計需求（Wireframe Level）

#### 互動需求

### P2. 首頁 Home
#### 頁面目的
作為所有狀態的主入口，依使用者當前狀態展示不同內容。

**顯示內容**

- 主 CTA：「設定押金」

**互動**
- 點擊後進入 P3 押金選擇頁

#### 狀態 B：已押金、未設目標
**顯示內容**
- 押金已確認卡（顯示金額）
- 解鎖成功提示
- 主 CTA：「建立我的運動目標」

#### 狀態 C：已有進行中目標
**顯示內容**
- 目標摘要卡
  - 目標名稱
  - 運動類型
  - 完成進度
  - 截止日期
  - 狀態標籤：進行中
- 寵物成長卡
  - 寵物圖像
  - 成長階段
  - 成長描述
- 快捷入口
  - 去打卡
  - 看進度
  - 看寵物收藏

#### 狀態 D：有歷史完成 / 失敗目標
**顯示內容**
- 最近一次結果卡
- 歷史目標列表入口

- Mobile First 單欄設計
- 首屏需清楚顯示主要 CTA
- 重要狀態需用顏色 / icon / label 區分

  - 300 元
- CTA：「確認押金」

#### 互動需求
- 預設不可直接送出，需先選擇金額
- 選擇後按鈕變為可點擊
- 點擊 CTA 進入 P4

#### 狀態需求
- 未選擇
- 已選擇
- 點擊確認後 loading / 成功

- 顯示押金金額
- 說明文字

- CTA：「開始設定目標」

- 點擊後進入 P5 目標設定頁

### P5. 目標設定頁
#### 頁面目的
讓使用者建立一個具體可追蹤的運動目標。

#### 建議欄位
- 目標名稱
- 運動類型
- 每週頻率
- 單次時長
- 目標週期
- 開始日期

- 結束日期不可早於開始日期

#### 頁面目的

#### 主要模組

  - 運動類型
  - 期間
  - 進度條
  - 已完成 / 目標次數

  - 進行中 / 已完成 / 失敗

- 進行中
- 已完成

- 尚未獲得任何寵物：空狀態

---

### P9. 完成 / 失敗結果頁（或彈窗）

- 本次目標未完成
- 押金未返還
- 可重新開始下一個目標
- CTA：「重新設定新目標」

## 十一、成功指標

### 11.1 Prototype 驗證指標
- 80% 以上測試者能在第一次使用時理解「先押金，後設目標」規則
- 85% 以上測試者可順利完成押金到目標建立流程
- 80% 以上測試者能明確辨識三種目標狀態：進行中 / 已完成 / 失敗
- 75% 以上測試者能理解寵物成長與目標進度的關係
- 75% 以上測試者能在進度頁中理解自己「有變化」的資訊呈現

### 11.2 正式產品後續可追蹤指標（非本次驗收）
- 7 日留存率
- 30 日目標完成率
- 押金設定後的目標建立率
- 第一個目標完成後再次建立第二個目標的比例
- 進度頁使用率
- 寵物收藏頁回訪率

## 十二、驗收標準

### 12.1 押金機制
- 使用者未設定押金前，不可進入目標設定流程
- 押金金額僅可選 100 / 200 / 300 元
- 未選擇金額時，確認按鈕不可點擊
- 押金確認成功後，目標設定功能需被明確解鎖
- 首頁需能正確顯示目前押金狀態

### 12.2 目標設定
- 使用者必須在押金確認後才能建立目標
- 目標建立頁需能完成基本表單輸入
- 必填欄位未填時，不可送出
- 建立成功後可在首頁或目標詳情頁看到新目標

### 12.3 目標狀態
- 系統需支援三種狀態呈現
  - 進行中
  - 已完成
  - 失敗
- 三種狀態在列表卡片與詳情頁需有明顯區別
- 20 筆目標 mock data 需可正確展示不同狀態

### 12.4 打卡與進度
- 使用者可從進行中目標執行打卡
- 打卡後進度數值需更新
- 打卡後寵物成長狀態需同步更新
- 若打卡後達成目標條件，需切換為完成狀態

### 12.5 完成結果
- 目標完成後需顯示完成結果頁或完成彈窗
- 完成結果需包含
  - 完成訊息
  - 新寵物獲得
  - 押金返還成功
- 完成的寵物需可在收藏頁中看到

### 12.6 失敗結果
- 目標失敗後需顯示失敗狀態
- 失敗結果需明確顯示押金不返還
- 使用者需可從失敗狀態重新開始新的押金設定流程

### 12.7 進度頁
- 需可看到至少一組身體變化摘要資料
- 需可看到進度統計與趨勢視覺
- 所有內容需以 mock data 正常顯示

### 12.8 寵物收藏頁
- 若有完成目標，收藏頁需顯示已解鎖寵物
- 若尚未完成任何目標，需有空狀態頁
- 寵物成長中與已解鎖狀態需清楚區分

### 12.9 Mobile First
- 所有頁面需以手機介面為優先
- 不依賴桌機寬版設計才能完成主流程
- 重要 CTA 在首屏或容易觸及的位置

## 十三、Corner Case

### 13.1 流程限制類
- 使用者未設定押金，卻嘗試從其他入口進入目標設定頁
- 使用者未選擇押金金額就點擊確認
- 使用者押金成功後，未建立目標就離開頁面
- 使用者已有進行中目標，又再次想建立新目標

### 13.2 表單與資料類
- 目標名稱過長
- 頻率為 0 或空值
- 結束日期早於開始日期
- mock data 為空時頁面需顯示空狀態
- 20 筆列表在手機上需能正常滾動，不應破版

### 13.3 狀態切換類
- 打卡後剛好完成最後一次目標時，需正確進入完成狀態
- 目標到期但尚未達標，需正確進入失敗狀態
- 已完成目標不可再繼續打卡
- 已失敗目標不可誤顯示為返還押金

### 13.4 收藏與進度類
- 尚未獲得任何寵物時的收藏空狀態
- 寵物收藏很多時的列表瀏覽
- 身體變化頁無資料時的預設提示
- 完成目標但尚未進收藏頁查看時，首頁仍需能提示有新寵物

### 13.5 Demo 特有情境
- 使用者返回上一頁再進入，頁面狀態需合理
- Demo 中的押金與退款文案需明確註記為模擬流程，避免誤解為真實交易
- 若展示過程需切不同狀態，前端需能快速切換 mock 狀態供 demo 使用

## 十四、備註 / 假設

### 14.1 合理假設
以下因原始 Spot 未明確定義，先做合理假設：

1. **同一時間僅支援一個進行中目標**
   - 目的：簡化押金、目標、寵物的一對一關係
   - 歷史上可有多個已完成 / 失敗目標

2. **每一個目標對應一隻寵物**
   - 目標進行中：寵物成長
   - 目標完成：寵物解鎖進收藏
   - 目標失敗：該次不獲得寵物

3. **目標失敗條件**
   - 假設為「截止日期到達時仍未完成 `targetCount`」

4. **運動打卡為手動自填**
   - 本次 prototype 不驗證真實運動行為

5. **押金與退款皆為 demo 流程**
   - 不涉及實際金流扣款與退款

6. **使用者已處於可使用 App 狀態**
   - 本次不處理登入 / 註冊流程

7. **身體變化資料為視覺化展示用途**
   - 不作為醫療、健康專業判讀依據

### 14.2 設計提醒
- 流程核心規則必須非常明確：**先押金，後設目標**
- 狀態切換必須清楚：**進行中 / 已完成 / 失敗**
- 完成回饋要夠強，因為這是產品情感高峰
- 失敗回饋不可過度懲罰式，但需清楚表達押金未返還
- UI 需優先支援 mobile first 的單手操作與首屏理解
